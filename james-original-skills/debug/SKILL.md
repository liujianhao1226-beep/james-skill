# Debug Skill

系统化定位根因，不盲猜。

## 核心原则

**先复现，再分析，最后修。** 永远不要在没理解根因的情况下修代码——那样只是把症状藏起来。

## 工作流

### Step 1：理解问题（1 分钟内完成）

按顺序问自己：
1. **错误信息是什么？** —— 完整读一遍，不要略过
2. **期望行为是什么？** —— 用户觉得应该发生什么
3. **实际行为是什么？** —— 和期望的差异
4. **这个问题是一直出现，还是偶发？**

如果用户还没给错误信息，引导粘贴：`请把完整的报错信息发给我（截图或文字都可以）`

### Step 2：定位根因

**假设驱动调试**——不要线性扫代码，而是：

1. **列出所有可能的根因**（通常 3-5 个）
2. **每个根因对应一个验证方法**：看哪行代码、看哪个日志、加哪句打印
3. **按可能性排序**，从最高开始验证
4. **每验证一个，缩小范围**，直到剩下唯一根因

常用定位手段（按场景选用）：
| 场景 | 手段 |
|---|---|
| 有报错堆栈 | 从底向上读堆栈，找第一行自己项目的代码 |
| 无报错但行为异常 | 二分注释法（注释一半代码看是否恢复）|
| 偶发性问题 | 问触发条件：什么操作/数据/时机必现 |
| 性能问题 | `console.time` / `time.time` / profiler |
| 并发/竞态 | 查锁/事务/共享状态，梳理执行时序 |

### Step 3：给出修复方案

分三层：

**立即修复**：直接给出代码改动，解释为什么这样修  
**预防建议**：同类问题如何避免（如加校验、加日志）  
**测试建议**：怎么验证修复是成功的

### Step 4：复盘（如果用户愿意）

如果问题有价值，引导思考：
- 这个 bug 暴露了测试的哪个盲区？
- 如何加一个用例防止这类问题重现？

## 常见问题速查

| 现象 | 可能根因 | 快速检查 |
|---|---|---|
| `undefined is not a function` | 函数未定义/导入错误/变量名拼写 | 检查 import / 函数声明 |
| 页面白屏 | JS 报错/资源加载失败/路由错误 | F12 看 Console |
| API 返回 500 | 后端异常未捕获/参数缺传 | 看服务端日志 |
| `Cannot read property 'x' of null` | 取值前未判空 | 找所有 `.x` 调用，加 `?.` 或判空 |
| 数据不更新 | 状态未刷新/缓存/请求未发 | Network 面板看请求是否发出 |
| 偶现 bug | 竞态条件/异步时序/外部数据依赖 | 找所有 setTimeout/async/并发写 |

## 语气

- 不要说「这代码有问题」——说「这里可能是根因」
- 不确定时说「让我验证一下」，不要盲猜
- 修完问一句「现在正常了吗」确认效果
