# 实际风格示例

下面的例子用于对齐“从导图路径到真实用例表”的写法。

## 示例 1：基础 happy path

### 输入路径
`组网管理 -> HAC组网 -> HAC DHCP开局 -> HAC完成配置向导 -> 再接入AP -> 有线组网成功`

### 输出用例
- 所属模块：`/组网管理/HAC组网`
- 用例标题：`【HAC DHCP开局】HAC完成配置向导，再接入AP-有线组网成功`
- 前置条件：
  1. HAC 的 WAN 口连接联网的 DHCP 服务器
  2. 测试设备类型：HAC+GTW06B+GTW05B+GTW04A
- 步骤：
  1. HAC 复位后完成配置向导
  2. 将初始化的 GTW06B+GTW05B+GTW04A 接入 HAC
  3. 终端连接 AP 的无线网络
- 预期：
  1. 管理页面可查看 3 台 AP 自动上线
  2. 终端可以正常上网
- 优先级：`1`
- 用例类型：`功能测试`
- 适用阶段：`版本验证阶段`

## 示例 2：状态迁移需要拆例

### 输入路径
`组网管理 -> HAC组网 -> HAC DHCP开局 -> HAC完成配置向导 -> HAC重启 -> 再接入AP -> 有线组网成功`

### 转换要点
- `HAC重启` 是状态迁移，不应被忽略
- 这条路径应拆为独立用例，不要并入基础 happy path

### 建议标题
`【HAC DHCP开局】HAC完成配置向导，HAC重启后再接入AP-有线组网成功`

## 示例 3：不同触发方式要拆例

### 输入路径 A
`组网管理 -> HAC组网 -> 页面无线组网 -> 页面点击添加 -> 无线组网成功`

### 输入路径 B
`组网管理 -> HAC组网 -> 按键无线组网 -> 弱电AP按键 -> 无线组网成功`

### 规则
尽管结果都是“组网成功”，但页面方式与按键方式验证目标不同，应拆成两条用例。

## 示例 4：机型枚举不应默认展开

如果路径只是写成：

`AC组网 -> 并联组网 -> 上网方式 dhcp -> GTW06A/GTW05A/GTW04A -> 2VAC+组网AP -> 可以组网`

默认做法：
- 把机型组合写进前置条件
- 生成 1 条代表用例

只有当思维导图明确要验证“某型号兼容、某型号不兼容”时，才按型号拆多条。

## 示例 5：版本矩阵不是设计用例

如果原 CSV 后面跟了很多 `V1.4.x / PASS / FAIL / NA` 列：
- 不要把这些列重新拆成设计用例
- 这些列属于执行记录，不属于前 8 列设计主体

## 示例 6：图片先转路径，再转用例

图片里看到的分支如果是：

- 组网管理
  - HAC组网
    - HAC DHCP开局
      - HAC完成配置向导
        - 再接入AP
          - 有线组网成功

正确做法是先整理成路径：

`组网管理 -> HAC组网 -> HAC DHCP开局 -> HAC完成配置向导 -> 再接入AP -> 有线组网成功`

再按字段拆到测试用例表。

## 示例 7：IoT 配网场景——网络类型合并

### 输入路径
```
功能 -> 配网 -> 手机网络连接 -> 手机4G网络，发起配网
功能 -> 配网 -> 手机网络连接 -> 手机5G网络，发起配网
功能 -> 配网 -> 手机网络连接 -> 手机局域网络，发起配网
功能 -> 配网 -> 手机网络连接 -> 手机外网网络，发起配网
```

### 转换要点
- 4G/5G/局域网/外网的配网流程完全一致，不应拆成 4 条独立用例
- 合并为 1 条主用例，网络类型写入前置条件

### 输出用例
- 所属模块：`/功能/配网`
- 用例标题：`【配网】不同手机网络环境下发起配网-配网成功`
- 前置条件：
  1. 设备已进入配网状态
  2. 手机网络环境：分别验证 4G / 5G / 局域网 / 外网
- 步骤：
  1. 手机在指定网络环境下打开 App
  2. 发起设备配网流程
  3. 输入正确的 WiFi 密码完成配网
- 预期：
  1. 配网成功，设备在 App 中显示在线
- 优先级：`1`
- 用例类型：`功能测试`
- 适用阶段：`版本验证阶段`

## 示例 8：IoT 配网异常——密码/SSID 边界

### 输入路径
```
功能 -> 配网 -> 密码异常 -> 错误密码
功能 -> 配网 -> 密码异常 -> 空密码/无密码的正常和错误入网
功能 -> 配网 -> 密码异常 -> 密码长度上限的正常和错误入网
功能 -> 配网 -> 密码异常 -> 密码带特殊字符的正常和错误入网
```

### 转换要点
- 每条路径的验证目标不同（错误密码、空密码、长度上限、特殊字符），应保留为独立用例
- 父节点"密码异常"作为场景标签

### 输出用例（其中 1 条）
- 所属模块：`/功能/配网`
- 用例标题：`【密码异常】密码长度达到上限时正常和错误入网`
- 前置条件：
  1. 设备已进入配网状态
  2. WiFi 密码设置为最大长度（63 字符）
- 步骤：
  1. 输入正确的最大长度密码，发起配网
  2. 输入超过最大长度的密码，发起配网
- 预期：
  1. 正确密码配网成功
  2. 超长密码被截断或提示错误，配网失败
- 优先级：`2`
- 用例类型：`边界测试`
- 适用阶段：`版本验证阶段`

## 示例 9：IoT 语音控制

### 输入路径
`功能 -> 语音识别 -> 语音控制灯光 -> 开灯/关灯指令 -> 灯光状态正确切换`

### 输出用例
- 所属模块：`/功能/语音识别`
- 用例标题：`【语音控制】语音指令开灯关灯-灯光状态正确切换`
- 前置条件：
  1. 设备已配网并在线
  2. 语音助手已启用
- 步骤：
  1. 语音唤醒设备
  2. 下发"开灯"指令
  3. 确认灯光开启后，下发"关灯"指令
- 预期：
  1. 开灯指令后灯光在 2 秒内开启
  2. 关灯指令后灯光在 2 秒内关闭
  3. App 端状态同步更新
- 优先级：`1`
- 用例类型：`功能测试`
- 适用阶段：`版本验证阶段`

## 示例 10：IoT 多端控制冲突

### 输入路径
`压力测试 -> 多端同时控制冲突`

### 转换要点
- 路径较短，但"多端同时控制冲突"是高价值场景，需要补充具体步骤
- 应结合产品实际控制端（App、屏幕、语音、物理按键）展开

### 输出用例
- 所属模块：`/压力测试`
- 用例标题：`【多端冲突】App和本地屏幕同时操作-设备状态最终一致`
- 前置条件：
  1. 设备已配网并在线
  2. App 和本地屏幕均可控制设备
- 步骤：
  1. 在 App 端发起开灯操作
  2. 同时在本地屏幕端发起关灯操作
  3. 等待 3 秒后检查设备实际状态
- 预期：
  1. 设备最终处于确定状态（开或关），不出现闪烁或无响应
  2. App 端和本地屏幕端状态显示一致
- 优先级：`2`
- 用例类型：`场景测试`
- 适用阶段：`版本验证阶段`

## 示例 11：IoT 注释节点处理

### 输入路径
```
功能 -> 配网 -> 路由异常 -> 离线设备，更换路由
  └── 注：考虑App对离线设备有重新发起连接的入口
```

### 转换要点
- "注：..."是注释节点，不生成独立用例
- 注释内容应融入相关用例的前置条件或步骤中

### 输出用例
- 所属模块：`/功能/配网`
- 用例标题：`【路由异常】离线设备更换路由后重新连接`
- 前置条件：
  1. 设备已配网但当前处于离线状态
  2. 原路由已不可用
- 步骤：
  1. 更换新路由并确保网络正常
  2. 在 App 中找到离线设备，点击重新连接入口
  3. 按提示完成重新配网流程
- 预期：
  1. 设备成功连接新路由，状态变为在线
  2. 设备原有配置和绑定关系保留
- 优先级：`2`
- 用例类型：`功能测试`
- 适用阶段：`版本验证阶段`

## 示例 12：IoT 屏幕交互——不同手势保留拆分

### 输入路径
```
功能 -> 屏幕交互 -> 左右横滑，循环切换
功能 -> 屏幕交互 -> 主页面向下滑动可呼出设置页
功能 -> 屏幕交互 -> 快速连续滑动不会导致页面错乱或崩溃
```

### 转换要点
- 不同手势/交互方式验证目标不同，应保留为独立用例
- "快速连续滑动"属于压力/稳定性测试，用例类型应标为 `场景测试`

### 输出用例（第 3 条）
- 所属模块：`/功能/屏幕交互`
- 用例标题：`【屏幕交互】快速连续滑动-页面不错乱不崩溃`
- 前置条件：
  1. 设备已开机并进入主页面
- 步骤：
  1. 在主页面快速连续左右滑动 20 次以上
  2. 观察页面切换是否正常
  3. 检查是否出现白屏、黑屏、卡死或崩溃
- 预期：
  1. 页面切换流畅，无错乱
  2. 无白屏、黑屏、卡死或崩溃现象
- 优先级：`2`
- 用例类型：`场景测试`
- 适用阶段：`版本验证阶段`
