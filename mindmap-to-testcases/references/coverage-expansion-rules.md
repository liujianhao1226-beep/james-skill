# 路径展开与规则补齐

本 skill 的默认策略不是"叶子节点直转"，而是：

1. 每条叶子路径先生成 1 条主用例
2. 再按规则补齐高价值扩展
3. 最后去重和压缩

## 1. 主用例

每条满足"有动作 + 有结果"的路径，至少生成 1 条主用例。

主用例通常是：
- happy path
- 或者路径本身已经明确写出的失败/不支持场景

## 2. 自动补齐规则

### A. 异常 / 反向补齐

遇到下列信号时，优先补异常用例：
- 失败、不支持、不可、无法、报错、拒绝
- 重复、超时、越权
- 已绑定 / 已存在 / 已组网
- 密码错误、密码为空、SSID异常
- 配网中断、升级中断、连接断开

补齐原则：
- 优先补"最直接的反向路径"
- 只补 1 到 2 条最有价值的异常，不做词语级穷举
- 异常用例默认优先级 `2`，长尾异常可降为 `3`

### B. 边界补齐

只有当路径中存在明确的范围/数量/限制信号时才补边界，例如：
- 数量：1台、2台、15秒内、最大/最小
- 范围：首尾、上下界、长度、容量
- 特殊值：空值、0、负值、极大值
- 密码长度上限、SSID长度上限、特殊字符

边界补齐原则：
- 至少考虑：边界内、边界值、超界
- 不是所有路径都强制补边界
- 边界用例通常优先级 `2` 或 `3`

### C. 状态迁移补齐

遇到下列信号时，优先补状态迁移用例：
- 重启后、复位后、断电后、重新接入
- 完成配置向导后、切换前后、回退
- 一起上电、OTA升级后、恢复出厂后
- 在线→离线、离线→在线、配网→已绑定

状态迁移用例默认优先级 `2`；如果它验证的是配置持久化、核心恢复能力，可升到 `1`。

### D. 场景 / 端到端补齐

满足以下特征时，补场景测试：
- 跨模块
- 多步骤串联明显
- 页面操作 + 设备状态 + 上线/联网结果同时出现
- 用户故事完整（例如"完成配置向导 -> 接入 AP -> 登录页面检查 -> 联网验证"）

端到端场景通常优先级 `1` 或 `2`。

### E. 多端控制冲突补齐（IoT 特有）

当路径涉及设备控制时，检查是否需要补充多端冲突场景：
- App 和本地屏幕/按键同时操作
- App 和语音同时下发指令
- 多个 App 用户同时控制同一设备
- 自动化/定时任务与手动操作冲突

补齐信号：
- 路径中出现"控制""开关""调节""设置"等动作词
- 路径中出现"App""语音""屏幕""按键""旋钮"等控制端词
- 路径中出现"定时""自动化""联动"等自动化词

多端冲突用例默认优先级 `2`，涉及安全态（如灯光异常闪烁、设备不响应）可升到 `1`。

### F. 语音交互补齐（IoT 特有）

当路径涉及语音控制时，检查是否需要补充：
- 语音唤醒失败 / 误唤醒
- 噪声环境下语音识别
- 连续语音指令
- 语音指令与当前设备状态冲突（如设备离线时下发语音指令）
- 方言 / 语速 / 音量边界

语音用例默认优先级 `2`。

### G. OTA / 固件升级补齐（IoT 特有）

当路径涉及 OTA 或固件升级时，检查是否需要补充：
- 升级过程断电 / 断网
- 升级后配置持久化验证
- 升级后功能回归验证
- 版本回退
- 升级进度显示和提示

OTA 相关用例默认优先级 `1` 或 `2`（升级中断为 `1`）。

### H. 屏幕/UI 交互补齐（IoT 特有）

当路径涉及屏幕交互时，检查是否需要补充：
- 快速连续滑动 / 连点稳定性
- 多点触控 / 斜滑
- 屏幕旋转 / 横竖屏切换
- 长时间显示（烧屏风险）
- 屏保触发和退出
- 弱光 / 强光环境下显示

屏幕交互用例默认优先级 `2` 或 `3`，压力/稳定性相关可升到 `2`。

## 3. 扩展数量控制

默认控制在：
- 1 条主用例
- 外加 0 到 3 条高价值扩展

不要在单条路径上同时展开十几个衍生用例。

IoT 特有扩展（E/F/G/H）同样受此限制，优先补最高价值的 1-2 条。

## 4. 去重与收敛

### 机型维度
默认只在以下情况下拆开：
- 节点明确表示"兼容性/适配性"验证
- 失败只发生在特定型号
- 型号差异本身影响链路、协议、能力边界

否则把机型组合作为 `前置条件`，不要拆成同逻辑多条用例。

### 版本维度
下列内容默认不展开为设计用例：
- 纯版本号
- 纯 build 号
- 执行人或测试轮次列
- PASS/FAIL/NA 结果列

### 网络类型维度
下列情况默认合并为 1 条用例，网络类型写入前置条件：
- 手机4G/5G网络发起配网（流程完全一致）
- 局域网/外网控制（流程完全一致）

只有当网络类型本身影响测试结果时才拆开（如 5G WiFi 不支持某设备配网）。

## 5. 优先级与用例类型建议

| 触发信号 | 默认优先级 | 默认类型 |
| --- | --- | --- |
| 主流程 / happy path | 1 | 功能测试 |
| 关键状态迁移 | 1 或 2 | 功能测试 |
| 典型失败 / 不支持 / 拒绝 | 2 | 异常测试 |
| 明确边界 / 上下界 / 数量限制 | 2 或 3 | 边界测试 |
| 多模块串联 / 端到端 | 1 或 2 | 场景测试 |
| 多端控制冲突 | 2 | 场景测试 |
| 语音交互异常 | 2 | 异常测试 |
| OTA 升级中断 | 1 | 异常测试 |
| OTA 升级后回归 | 1 或 2 | 功能测试 |
| 屏幕压力/稳定性 | 2 或 3 | 场景测试 |

## 6. 最终判断标准

只有同时满足下面三个条件的扩展，才值得保留：
- 明显提升覆盖价值
- 用户或导图信息可以支持它成立
- 可以写成可执行、可验收的步骤和预期
