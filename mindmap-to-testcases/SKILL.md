# Mindmap To Testcases

将思维导图、XMind、FreeMind `.mm`、Markdown 大纲、文本树、CSV 路径导出或脑图截图，转换成可执行的测试用例表。默认输出的是"设计用例主体"，固定 8 列：`所属模块`、`用例标题`、`前置条件`、`步骤`、`预期`、`优先级`、`用例类型`、`适用阶段`。

默认离线工作，只使用当前对话里的文件和图片，不假设 Jira、TestRail、飞书多维表或其他连接器可用。除非用户明确要求，否则不要生成执行结果列、PASS/FAIL 列或版本执行记录列。

中文输入默认输出中文字段名和中文字段值。

对 IoT / 智能硬件领域有专门优化：配网、语音控制、屏幕交互、OTA升级、传感器、多端控制冲突、产测模式、断电恢复等场景均有对应的节点映射规则和覆盖扩展策略。

## When To Use

- 用户要把脑图、测试范围大纲、模块树或路径清单转成可执行测试用例。
- 输入是 `.xmind`、`.mm`、`.md`、`.txt`、`.csv`，或脑图截图。
- 用户希望在主用例之外补齐异常、边界、状态迁移、场景链路。
- 测试范围很大，适合按模块拆开并行处理。
- 用户希望由资深测试工程师视角再做一次缺口复评和优化。
- IoT / 智能硬件产品的系统测试范围大纲需要转化为用例表。
- 用户提到"配网""语音""屏幕交互""OTA""多端控制""产测"等 IoT 关键词。

不要用于：

- 已经是正式测试用例表，只需要微调个别字段。
- 需求还没成型，只在头脑风暴阶段。
- 用户要的是测试计划、测试报告、缺陷单，而不是测试用例。

## Default Workflow

1. 建立输入清单并识别输入类型。
   - 结构化脑图：`.xmind`
   - FreeMind / Freeplane：`.mm`
   - 文本树：`.md`、`.markdown`、`.txt`
   - 路径导出：`.csv`
   - 图片：截图、拍照、PNG、JPG、JPEG、WEBP
   - 已有测试用例表：通常只作为风格参考或对比基线，不直接当作脑图源
2. 先把输入统一成"根到叶路径"。
   - `.xmind`、`.mm`、`.md`、`.txt`、`.csv`：优先运行 `scripts/extract_mindmap_paths.py`（推荐加 `--strip-root`）
   - 图片：先用视觉能力还原层级，再整理成路径清单
   - 检查脚本输出的统计信息（路径总数、注释路径数、深度分布），判断规模和拆分策略
3. 识别产品领域和测试维度。
   - 如果是 IoT / 智能硬件产品，识别涉及的测试域：配网、远程控制、语音、屏幕交互、OTA、传感器、多端控制、产测、嵌入式等
   - 根据领域选择对应的节点映射规则和覆盖扩展策略
4. 对每条路径做节点分类。
   - 模块/功能域
   - 场景标签
   - 前置条件
   - 动作步骤
   - 预期结果
   - 兼容性、版本、数量、权限、网络、状态等维度
   - 注释节点（`is_annotation: true`）：不生成独立用例，融入相关用例
5. 先生成主用例，再按规则补齐高价值扩展。
   - 每条"有动作且有结果"的根到叶路径，至少生成 1 条主用例
   - 再补齐异常、边界、状态迁移、端到端场景
   - IoT 领域额外检查：多端控制冲突、语音交互、OTA中断、屏幕压力、断电恢复
6. 去重、压缩噪声、保留高价值覆盖。
   - 流程完全一致仅网络类型不同的路径（如4G/5G配网），合并为 1 条
   - 机型枚举不展开，写入前置条件
7. 校验结构和内容。
8. 用户要求文件时，先校验，再导出 CSV 或 XLSX。

## Input Normalization

优先使用脚本，不手抄长树：

```bash
python scripts/extract_mindmap_paths.py input.xmind normalized-paths.json
python scripts/extract_mindmap_paths.py input.mm normalized-paths.json
python scripts/extract_mindmap_paths.py input.md normalized-paths.json
python scripts/extract_mindmap_paths.py input.csv normalized-paths.json
```

推荐加 `--strip-root` 去掉文档标题根节点（如"XX系统测试范围大纲"），让路径从一级模块开始：

```bash
python scripts/extract_mindmap_paths.py input.mm normalized-paths.json --strip-root
```

脚本会自动输出路径统计信息（总数、注释路径数、深度分布），用于快速判断脑图规模和拆分策略。

### 注释节点处理

脚本会自动识别以 `注：`、`备注：`、`说明：`、`NOTE:`、`TODO:` 等前缀开头的节点，并在输出中标记 `is_annotation: true`。

处理原则：
- 注释节点不生成独立测试用例
- 注释内容如果包含测试约束或补充说明，应融入相关用例的前置条件或步骤中
- 空节点和 `[untitled]` 节点会被自动过滤，不出现在路径中

图片输入的处理要求：

- 先还原主干层级和测试意图，不追求逐像素抄录。
- 看不清的型号、版本、备注不要硬猜。
- 必须做最小合理补全时，要明确写出假设。

详细规则见：

- `references/input-normalization.md`
- `references/mindmap-node-mapping.md`

## Manual Baseline Alignment

如果用户除了思维导图之外，还提供了人工编写的测试用例表、历史版本用例表或评审通过的样例用例，不要直接按导图叶子路径机械展开到最终交付表。先把这类文件当作“粒度与风格基线”，再决定最终输出的收敛方式。

使用顺序：

1. 先抽样查看人工版 8 到 20 条用例，判断它偏向 `叶子直出`、`场景聚合`，还是 `检查清单聚合`。
2. 再把人工用例与导图路径做关系映射：
   - `1:1 直接映射`
   - `同父场景下兄弟叶子聚合`
   - `生命周期连续叶子聚合`
   - `默认值/检查项清单聚合`
   - `因触发方式或风险域不同而保留拆分`
3. 如果人工版明显更偏场景化，最终交付优先对齐人工版粒度；完整叶子展开稿只作为中间稿或附录保留。
4. 模块名、优先级、用例类型、适用阶段，优先参考人工版的稳定写法，而不是只按导图原路径或叶子关键词推断。

特别注意：

- `不能配网及配对`、`离线模式` 这类词，不一定天然属于“异常测试”；要先看它在父场景里是主流程、限制校验，还是失败分支。
- `默认状态`、`检查默认状态`、`启动完成` 这类父节点，常常更适合生成 1 条检查清单式或端到端式用例，而不是一叶一例。
- `左右滑动`、`下拉呼出`、`点击反馈`、`连续滑动压力`、`多点触控`、`烧屏` 这类不同交互模式或风险域，通常应保留为独立用例，不要过度合并。

详细规则见：

- `references/manual-alignment.md`

## Multi-Agent Merge And Alignment

如果按模块让多个 sub-agent 并行产出候选用例，不要手工复制粘贴合并。优先让每个 agent 输出固定 JSON 结构，再用脚本统一做清洗、去重、模块均衡和限量收敛。

推荐命令：

```bash
python scripts/align_testcases.py agent1.json agent2.json agent3.json -o aligned.json --limit 100
```

这个脚本负责：

- 统一字段名与字段格式
- 去掉标题中的生成痕迹
- 按 `所属模块 + 用例标题` 去重
- 在设置 `--limit` 时按顶层模块均衡保留
- 产出可继续交给 `validate_testcases.py` 和 `render_testcases.py` 的标准 JSON

## Module Split And Sub-agents

当测试范围足够大时，不要由主 agent 顺序硬写全量用例。优先按模块拆分，并用多个 sub-agent 并行处理不同模块。

适合拆模块的信号：

- 一级或二级模块达到 3 个以上，且模块之间相对独立。
- 根到叶路径很多，单线程展开容易遗漏。
- 用户明确要求“分模块并行”或“让资深测试工程师补缺口”。
- 存在明显不同的测试域，例如：配网、远程控制、语音、AI、OTA、性能、稳定性、产测。

不适合拆模块的场景：

- 脑图很小，全部内容不到 20 到 30 条路径。
- 模块间强耦合，必须先统一状态机或权限规则才能写用例。
- 用户只要一个小模块，不需要并行化。

### Sub-agent Role

每个 sub-agent 都扮演“资深高级测试工程师”，不是机械扩写器。它的职责是：

- 负责自己模块的路径理解、用例生成、缺口复评和优化。
- 基于脑图内容先产出主用例，再从测试设计角度主动补齐高价值缺口。
- 重点检查异常流、边界值、状态迁移、一致性、弱网/断电/重启、权限、恢复链路。
- 标出需要假设的地方，不擅自捏造需求。
- 不改其他模块，不和其他 sub-agent 争抢所有权。

### Recommended Split Strategy

模块拆分优先按“测试对象或风险域”划分，而不是平均切块。推荐顺序：

1. 先按一级模块拆，如 `/功能`、`/性能`、`/稳定性`、`/兼容性`。
2. 一级模块过大时，再按二级模块拆，如 `/功能/配网`、`/功能/远程控制`、`/功能/语音识别`。
3. 质量属性类模块可以单独拆，如 OTA、性能、稳定性、产测。
4. 跨模块一致性问题不要提前拆散，留给主 agent 做统一复核。

### Sub-agent Prompt Pattern

给每个 sub-agent 的任务要明确到模块和产物，不要笼统地说“帮我生成测试用例”。推荐包含这些约束：

- 你是资深高级测试工程师。
- 你只负责指定模块。
- 你先基于脑图路径生成主用例。
- 然后对该模块做复评，补充高价值缺口，但不要无限发散。
- 输出结构化 JSON，用固定字段名。
- 说明你补了哪些脑图原文之外但高价值的点。
- 不要覆盖别的模块，不要把共享规则改写成和主 agent 冲突的定义。

### Expected Sub-agent Output

每个 sub-agent 至少返回两部分：

1. 该模块的结构化用例集。
2. 该模块的“复评补充说明”，内容包括：
   - 发现的高价值缺口
   - 新增原因
   - 假设点和风险点

如果模块信息不足，sub-agent 要明确说“因需求缺失未展开”的原因，而不是编造流程。

### Main Agent Merge Rules

主 agent 负责统一收口，不把 sub-agent 的结果原样拼接就结束。必须做：

- 合并多个模块 JSON。
- 按 `所属模块 + 用例标题` 去重。
- 统一字段名、优先级、用例类型、适用阶段。
- 复核跨模块一致性问题。
- 单独保留“脑图原始展开”和“资深测试复评补充”的差异说明。

跨模块重点复核这些问题：

- 多端状态同步（App、本地屏幕、语音控制后各端状态是否一致）
- 本地控制与 App/语音冲突裁决
- 断电、重启、OTA 后配置持久化
- 远程控制弱网重试和最终一致性
- 安全态恢复和故障保护
- 配网状态与设备控制状态的衔接（配网成功后是否可立即控制）
- 定时任务/自动化与手动操作的优先级冲突
- 产测模式与正常模式的隔离和切换

## Testcase Generation Rules

### 所属模块

- 使用稳定模块路径，例如：`/组网管理/HAC组网`
- 不把设备型号、版本号、执行结果列写进 `所属模块`

### 用例标题

推荐模式：

`【场景标签】谁在什么条件下做什么，预期什么结果`

标题里应保留：

- 场景标签
- 关键动作
- 结果导向

不要在标题里保留来源痕迹，例如：

- “导图映射”
- “脑图展开”
- “原路径”
- “根据节点”

标题应直接表现为最终交付给测试团队使用的成品用例标题，而不是生成过程说明。

### 前置条件

只写静态条件，例如：

- 网络环境
- 设备拓扑和型号组合
- 账号/家庭/权限关系
- 数据状态
- 版本依赖

不要把执行动作写进前置条件。

### 步骤

- 保持 2 到 5 步优先
- 每步一个主动作
- 状态切换顺序明确
- 不要把预期混进步骤
- 不要出现“按导图路径进入”“覆盖导图中描述的条件”“根据脑图节点执行”这类过程痕迹
- 步骤要直接描述测试人员应执行的动作，而不是解释该动作来自哪里

### 预期

只写可观察结果，优先写：

- 页面或后台状态
- 联网/绑定/同步结果
- 文案、告警、错误提示
- 配置是否生效
- 设备是否进入正确状态

不要在预期中写：

- “满足导图要求”
- “与脑图一致”
- “符合原始路径”

预期应直接写成系统可观察结果，让读者无需知道脑图原文也能独立执行和验收。

### 优先级

默认使用 `1/2/3`：

- `1`：核心主流程、关键兼容链路、关键安全链路
- `2`：重要分支、典型异常、关键状态迁移
- `3`：长尾边界、低频异常

### 用例类型

只使用这些标准值：

- `功能测试`
- `异常测试`
- `边界测试`
- `场景测试`

### 适用阶段

默认 `版本验证阶段`。只有用户明确指定为冒烟、回归、提测、上线验收等阶段时才替换。

## Structured JSON Format

导出 CSV 或 XLSX 前，先组织成 JSON，再校验和渲染。推荐结构：

```json
{
  "cases": [
    {
      "所属模块": "/组网管理/HAC组网",
      "用例标题": "【HAC DHCP组网】HAC完成配置向导后再接入AP-有线组网成功",
      "前置条件": [
        "HAC 的 WAN 口连接 DHCP 网络",
        "测试设备组合为 HAC + GTW06B + GTW05B + GTW04A"
      ],
      "步骤": [
        "HAC 复位后完成配置向导",
        "将初始化后的 GTW06B、GTW05B、GTW04A 接入 HAC",
        "终端连接 AP 的无线网络"
      ],
      "预期": [
        "管理页面可查看 3 台 AP 自动上线",
        "终端可以正常上网"
      ],
      "优先级": "1",
      "用例类型": "功能测试",
      "适用阶段": "版本验证阶段"
    }
  ]
}
```

脚本同时接受英文别名字段：`module`、`title`、`preconditions`、`steps`、`expected`、`priority`、`case_type`、`stage`，但最终导出应统一为中文列和值。

## Validation And Export

校验：

```bash
python scripts/validate_testcases.py input.json
```

导出 CSV：

```bash
python scripts/render_testcases.py input.json output.csv
```

导出 XLSX：

```bash
python scripts/render_testcases.py input.json output.xlsx
```

导出要求：

- 先校验再导出，不要跳过校验。
- CSV 必须使用 `UTF-8 BOM`。
- 导出后要能回读通过基本自检。
- 如果目标文件被占用，允许回退到 `*.fixed.csv` 或带时间戳的新文件名。

## Common Mistakes

- 把版本执行记录列误当成新的测试用例。
- 把设备型号矩阵无脑展开成大量低价值重复用例。
- 只抄叶子节点，不补状态迁移和异常流。
- 在标题、步骤、预期里残留"导图映射""按导图路径""满足导图要求"等生成痕迹，导致用例不像最终交付物。
- sub-agent 之间没有模块边界，导致重复生成或定义冲突。
- sub-agent 只做"扩写"，没有做资深测试视角的缺口复评。
- 主 agent 不做最终合并去重和跨模块一致性复核。
- 用例里出现 `?????`、`�` 或中英文类型混杂还直接导出。
- 注释节点（`注：`、`备注：`等）被当作正常路径生成了独立用例。
- 空节点或 `[untitled]` 节点没有被过滤，产生了无意义路径。
- 根节点（文档标题如"XX系统测试范围大纲"）出现在每条路径中，导致模块路径冗余。

### IoT / 智能硬件领域常见错误

- 手机4G/5G/局域网/外网配网流程完全一致，却拆成多条独立用例，造成大量冗余。
- 忽略多端控制冲突场景（App + 本地屏幕 + 语音同时操作）。
- OTA 升级只写了正常流程，没有补升级中断（断电/断网）和升级后回归验证。
- 语音控制只写了 happy path，没有补噪声环境、连续指令、设备离线时语音指令等异常。
- 屏幕交互只写了基本手势，没有补快速连续操作的压力/稳定性测试。
- 断电恢复场景缺失：没有验证断电后配置持久化、定时任务恢复、设备状态恢复。
- 产测模式和正常功能测试混在一起，没有独立模块划分。
- 传感器测试只写了正常触发，没有补边界值（距离远近、灵敏度阈值）和环境干扰。

### Additional Manual-Alignment Mistakes

- 同一个父场景下其实应该聚合成 1 条场景用例，却仍然机械按兄弟叶子拆出多条碎片。
- `默认状态`、`检查默认值`、`启动完成` 这类清单型节点被拆成过多独立用例，导致最终表格不像人工交付物。
- 只看叶子关键词就把用例判成 `异常测试` 或 `边界测试`，没有先看父场景的真实测试意图。
- 机械保留导图原路径当 `所属模块`，没有根据人工版或业务阶段做模块收敛。
